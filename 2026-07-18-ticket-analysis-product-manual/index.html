01 / O que é

Um parecer consultivo, não um veredito.

O produto pega a foto Before (registrada pelo técnico no chamado) e as fotos After (coletadas no checklist), envia pro Gemini com um prompt versionado, e devolve um JSON estruturado: confiança, leitura operacional, contexto. Cabe ao humano da operação decidir se fecha, reprova, ou pula etapas.

Escopo
SP

Manutenções no sistema Ruas. Atividades configuradas em analysis_activity_ids. Estabelecimentos com modelo de abrigo cadastrado.

Modelo
gemini

gemini-flash-latest fixado no serviço. REST via Laravel Http, PHP 7.4 compat. Zero SDK Google.

Resposta
JSON

Confiança 0–1, leitura operacional em pt-BR, contexto temporal/equipamento, alerta de incompatibilidade visual.

Decisão
humana

IA é consultiva. Não fecha chamado, não muda status, não pula etapas. Humano sempre decide.

Três princípios não negociáveis.

  • Idempotência por existência: UNIQUE(ticket_id). Reanalisar o mesmo ticket não duplica nem sobrescreve sem retry manual.
  • Reserva pré-dispatch: o slot no ledger é gravado antes do job ser despachado. Se o job falhar, o slot existe e a retentativa é consciente.
  • Default live versionado: nenhuma flag no .env.example. Dry-run exige override explícito em testes e gates técnicos.
O que NÃO faz

Limites assumidos.

  • Não reprova chamado. Reprovar é decisão do operador.
  • Não pula etapas. O fluxo oficial segue intacto.
  • Não persiste base64 de imagens. Mídia é resolvida on-the-fly.
  • Não expõe credenciais. .env.example idêntico a origin/main.
02 / Pipeline

Como funciona, do cron à resposta.

Cinco estágios, cada um com output verificável. Pode ser disparado pelo cron ou sob demanda pelo operador.

✓
01 · Seleção
Janela operacional [5h, 4h)

Command ticket-analysis:analyze-maintenance varre tickets completed com end_date na janela. Apenas tickets elegíveis passam: feature flag, sistema Ruas, status completed, activity_type_id configurado, estabelecimento confiável, sem análise prévia.

✓
02 · Reserva
Slot no ledger antes do dispatch

Para cada ticket elegível, reserva pré-dispatch grava a ticket_analysis em pending e os itens no ledger do run. A reserva é idempotente por UNIQUE(ticket_id).

✓
03 · Dispatch
Job assíncrono por slot

Para cada slot reservado, dispatcha TicketAnalysisJob com (analysis_id, expected_run_id). O job usa CAS por geração para evitar double-processing em redelivery.

✓
04 · Execução
Gemini live com prompt versionado

Resolver monta Before (tickets.filepath) e After (ticket_photos). ProfilePromptBuilder gera o prompt a partir de resources/ticket-analysis/profiles/maintenance.json. Gemini retorna JSON estruturado, validado pelo TicketAnalysisLiveResponseValidator.

✓
05 · Persistência
Status final + UI renderiza

Job marca completed_live com confiança, leitura operacional, contexto. UI AdminLTE inline na aba do ticket renderiza o parecer. Auditoria fica no ticket_analysis_runs + ledger de itens.

Mídia, zero base64

Como as fotos entram e saem.

Fluxo da imagem

Before: tickets.filepath (Drive) ↓ MaintenanceEvidenceMedia (resolve) ↓ Gemini REST (URL remota) ↓ Resposta JSON (sem payload binário) ↓ UI renderiza via URL

O banco nunca persiste base64. As URLs são resolvidas na hora do Gemini e na hora da renderização. Validação de mime e tamanho acontece no evidence resolver.

O que o prompt carrega.

Contexto fixo

SP Ruas · manutenção de abrigo - atividade: - shelter: - janela: 5h a 4h antes do end_date

Contexto variável

ticket_answers (checklist) - execution_start_date, execution_end_date - descrição do modelo de abrigo - regras visuais e temporais (v2)

Diretivas

- JSON estruturado (sem prosa) - confiança 0 a 1 - alerta se foto não bate com equipamento - pt-BR em todo o output

Anti-correções

- Não usar SLA fixo (removido) - Não inventar dados do chamado - Não expor completed_by_system

04 / UI

Como o operador usa.

AdminLTE inline na aba do ticket. Dois botões claros. Render responsivo desktop e mobile (390×844).

Cenário do operador
carregando…

Cenário 1: ticket sem análise.

Operador abre o ticket. Aba "Análise" mostra: "Este chamado ainda não possui análise por IA. Deseja gerar agora?"

Um botão: Executar análise. O job entra na fila, status vira pending, depois running, depois completed_live. UI faz polling do status.

Cenário 2: ticket com análise.

Operador abre o ticket. Aba "Análise" mostra o parecer completo: confiança, leitura operacional, contexto temporal, alerta de incompatibilidade (se houver).

Um botão: Abrir checklist e imagens. Read-only. Sem regenerar.
