Regras nao-negociaveis de William.

Extraidas do AGENTS.md de William e validadas pela auditoria adversarial. Estas 12 regras governam qualquer codigo saIA que entre no repositorio.

  1. 1
    PHP 7.4 apenas — zero recursos PHP 8.
    Nada de str_contains, match, named arguments, union types, mixed, enums, readonly, intersection types, never. ??= e arrow functions sao permitidos (sao 7.4).
  2. 2
    Laravel 8.x — nada de APIs 9/10/11.
    Nada de to_route(), novos metodos do query builder 9+, Attribute accessors. Use accessors/mutators old-style.
  3. 3
    Vue.js 3 + AdminLTE 3.
    Nada de Vue 2 lifecycle hacks. Nada de design system novo. Nada de React, Tailwind ou SPA.
  4. 4
    SOLID + Service Layer + Repository Pattern.
    Controllers orquestram, Repositories consultam, Policies autorizam. Camadas distintas.
  5. 5
    Evitar N+1. Usar chunk()/lazy() para datasets grandes.
    Nao basta eager load — processamento batch precisa de chunk.
  6. 6
    NAO rodar npm run watch/dev/production.
    Build commands exigem aprovacao explicita de William.
  7. 7
    WSL eh o UNICO ambiente de execucao.
    Comando padrao: wsl -d Ubuntu-22.04 -- bash -lc "cd /home/wpinheiro-dev-eletromidia/eletromidia/public_html/operacoes && ...". Nunca usar PowerShell para editar arquivos.
  8. 8
    Security gate: validar ambiente ANTES.
    Se WSL, Docker, Sail ou estado do git nao forem verificados, PARAR e perguntar a William.
  9. 9
    Economia de tokens: usar rg/sed para leituras focadas. Agrupar leituras independentes.
    Nao fazer dump de arquivos inteiros quando snippets bastam.
  10. 10
    Editar lock files livremente.
    composer.json, composer.lock, package.json, package-lock.json podem ser editados. Porem adicionar NOVOS pacotes segue regra de aprovacao da skill.
  11. 11
    Comandos preferidos: git status, git diff, php -l, composer update/install, npm install, npm audit, composer audit.
    Sempre lintar (php -l) e auditar antes de declarar pronto.
  12. 12
    AdminLTE 3 especificamente — nao AdminLTE 2.
    Bootstrap 4 permanece. A linguagem visual eh Blade + AdminLTE 3 + Bootstrap 4 + Vue 3 islands.

Roadmap de implementacao.

Fatiar para reduzir risco: provar evidencia e idempotencia, depois IA, depois tela, depois expansao.

P0. Fundacao sem Gemini

Config, migrations, models, repository, resolver, command, job dry-run e testes de espelhamento tardio.

P1. Analise real

Servico Gemini normalizado, prompt profiles versionados em JSON, normalizador de resposta e status needs_review.

P2. HITL

Resource, endpoint e tela Vue 3/AdminLTE 3 com fila, detalhe de ocorrencia e acoes (aprovar, rejeitar, escalar).

P3. Expansao

Novas atividades, estabelecimentos, RJ (se aprovado) e migracao do padrao AI de Posting para servico compartilhado.

P0: Tabelas e persistencia

  • saia_analysis_runs: execucao, escopo, contadores, modelo, versao de prompt, datas, erro agregado.
  • saia_analysis_items: unidade analisada, status, profile/version, input_hash, confidence, nota, resposta normalizada/bruta, evidencia, revisao humana.
  • Unique key: (analyzable_type, analyzable_id, profile_key, profile_version, input_hash).
  • Nao escrever estado saIA em tickets.

P2: Primeira tela HITL esperada

  • Fila de itens needs_review com filtros por estabelecimento, atividade e confianca.
  • Detalhe com imagens lado a lado, checklist e resposta IA em JSON legivel.
  • Acoes simples: aprovar, rejeitar, escalar, comentar.
  • Componente Vue 3 Options API + Vuex 4 namespaced module. Nenhuma acao automatica em ticket no MVP.

Decisoes para alinhar com William.

Recomendacao: SP + RJ com feature flag.
