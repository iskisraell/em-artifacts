A decisão fundadora: Radar como módulo nativo.

Em 5 de maio de 2026, a decisão foi tomada. O Radar não seria um app React separado com API-First. Seria um módulo dentro do Operações, usando o que já existia.

Cenário E — o escolhido

Radar como módulo nativo do Operações. Sessão, autorização, Blade, Vue/Vuex, Bootstrap/AdminLTE e Eloquent/repositories locais. Nada de proxy, nada de API separada, nada de auth customizada.

stack existentezero nova infra

O que NÃO fazer

  • Não adicionar React, Tailwind ou Zustand
  • Não criar app separado com auth própria
  • Não usar IndexedDB/PWA no cache
  • Não criar proxy/cache server separado
limites claros

Stack do Radar

Backend: Laravel 8 + PHP + MySQL 8 Frontend: Vue 3 + Vuex 4 + Laravel Mix Mapa: Leaflet + MarkerClusterer UI: Bootstrap 4 + AdminLTE Cache: Laravel Cache (Redis/file) Testes: PHPUnit + Pest

Restrições arquiteturais

✓ Sessão Operacoes — sem auth customizada ✓ Policies e permissions existentes ✓ Blade mount point para o Vue app ✓ Eloquent com repositories, não SQL raw ✓ CSP scope = hard backend boundary ✓ Request filters só estreitam, nunca ampliam

Cinco superfícies, um estado de filtro.

O mesmo payload de filtros alimenta lista, mapa, dashboard, detalhe e opções. Cada endpoint tem seu contrato — ninguém compartilha resource.

Lista de equipamentos

13 filtros, paginação infinita (24 por lote), Axios CancelToken para requests obsoletos. Cache key: scope + filters + page.

Resource

RadarCardResource → dados do place, mídia, ações → paginado, com has_more

Query budget

Página 1: 3-4 queries Append: 2 queries COUNT(*) só na página 1 ou quando filtros mudam

Único paginado

Mapa e dashboard não paginam. Agregam ou amostram. Só a lista usa paginação clássica com has_more.

Markers do mapa

Payload leve: id, lat, lng, status, alternative_code. Sem relações, sem contagens. Limite máximo de markers com enquadramento visual por escopo.

Leaflet + MarkerClusterer. Cache por escopo + filtros ativos. Sem recordsTotal — o contrato é de coordenadas, não de tabela.

Não é DataTable

Mapa não tem start, length, recordsFiltered. É outro contrato: markers + dados resumidos para popup. Reutilizar card resource quebraria o orçamento de queries.

Dashboard agregado

KPIs, charts e insights por concessão. Cache 300s (5 min). Não retorna rows individuais — retorna contagens, distribuições e tendências.

JSON público esconde arrays internos de IDs. O resource remove establishment_ids antes de serializar.

Stale-while-refresh

Endpoints pesados servem o último payload válido enquanto um request atualiza. Cold cache tem lock com espera curta e fallback limitado.

Detalhe do place

Buscado por place_id apenas quando o usuário abre um card ou clica em um marker. Tickets limitados a 50 recentes. Máquinas carregadas com status externo sob demanda.

faz

Links para Operações

Redireciona para /places/{id} e tickets existentes. Não cria ticket inline. Não duplica fluxos.

faz

Query budget: 4-6

LIMIT em tickets e machines. Abrir detalhe não recarrega lista, mapa ou dashboard.

Opções de filtro dinâmicas

O endpoint filter-options recomputa cada lista de opções a partir do universo já filtrado, excluindo o próprio campo. Cache 1800s (30 min).

Dropdowns refletem o que existe dentro do escopo autorizado + filtros ativos. Não são listas estáticas do Blade.

Cross-filtered

Selecionar um estabelecimento atualiza as praças disponíveis. Selecionar uma praça atualiza as filiais. O usuário sempre vê só o que ainda existe no recorte.

O cache aprendeu a respeitar concessão.

O risco não estava no usuário de uma praça só. Estava nos escopos multi-praça e nacional, onde limpar uma concessão podia não invalidar o cache de quem via várias.

Antes

Cache com rótulo genérico

O Radar sabia que o escopo era "nacional" ou "multi", mas não carregava as concessões reais na chave do cache.

Risco

Limpar uma praça não afetava todas

Payload nacional com dados de SP podia continuar stale mesmo depois de limpar o cache de SP. Cache genérico demais.

Agora

Chave leva as concessões reais

Single, multi e nacional dependem dos carimbos corretos. Se a concessão muda, o cache certo muda junto.

Clear por versão

radar:cache:clear --concession=sp troca só o carimbo da concessão escolhida. Não derruba cache de outras praças.

preciso

Warmup

radar:cache:warm --concession=sp --payload=all pré-aquece mapa, dashboard e filtros de uma concessão.

pronto

Health

radar:health --concession=sp confirma contagem, mapa e latência de cache sem mexer nas chaves de produção.

monitorado

Normalização de tipos

Um bug fazia work_zone_id[]=2&work_zone_id[]=1 e work_zone_id=1,2 produzirem hashes diferentes. Uma linha em flatArrayFilter() resolveu: array_map('strval', $values) antes de ordenar.

Stampede protection

Lock evita que múltiplas requisições recalculem o mesmo payload pesado ao mesmo tempo. Cold cache tem espera curta com fallback — o último payload válido é servido enquanto o novo é computado.

Por que payload-driven, e não Blade+DataTable.

O Operações usa Blade + DataTable. O Radar usa Vue + JSON payloads. Não foi acidente — foi resposta ao formato do produto.

Padrão nativo

Blade + DataTable

Blade renderiza components.card.filter. Inputs viram filter-<field>. DataTables serializa e o repository consome via withFilter($request). Simples para CRUD.

Radar

Vue + JSON payloads

Mount point Blade entrega URLs ao <radar-app>. Vuex envia query params para endpoints JSON. RadarController responde com envelope {response, status, message, options}.

1. Radar não é uma lista só
Tem cards, mapa, dashboard, detalhe e opções. DataTable resolve só a lista.
formato
2. Cada endpoint precisa de shape diferente
recordsTotal/recordsFiltered não serve para markers ou KPIs agregados.
contrato
3. Cache key precisa de contexto completo
radarCacheKey() hasheia scope + filters + payload. DataTable não tem esse conceito.
cache
4. Stale-request cancellation
Vuex cancela Axios calls antigas. Blade form submit não tem esse controle.
frontend
5. Filtros cross-filtered
Dropdowns atualizam contra o universo filtrado. Blade static options são mais simples, mas não servem.
UX
6. Mapa é stateful
Max markers, framing de escopo amplo, balanceamento multi-square. Não mapeia para pipeline de tabela.

O que pesa

Custo da escolha

  • Parece não-nativo dentro do Operações (bypassa Blade+DataTable)
  • Vocabulário duplicado (branch_ids vs shelter_model_id)
  • Filtros Radar-only não existem no Place filter scope
  • Filtragem de permissão reimplementada no RadarRepository
  • Qualquer rename toca Vue, Vuex, repository, cache, resources e testes
O que funciona

Benefício real

  • Um estado de filtro dirige 5 superfícies consistentemente
  • Cache keys estáveis com filtros normalizados em arrays
  • Dropdowns se adaptam ao resultado ativo (cross-filtered)
  • Vuex cancellation evita resposta antiga sobrescrever filtro novo
  • Cada endpoint devolve o shape que precisa, sem forçar DataTable

Regra que nunca pode quebrar

configuração da concessão ∩ permissões do usuário ∩ filtros da tela

Os filtros da tela só reduzem o resultado. Eles nunca dão acesso a uma praça que o perfil não tem. O escopo CSP é hard backend boundary — testado e coberto por RadarScopeBoundaryTest.
