O que aconteceu.

O Radar foi deployado para produção. Duas necessidades críticas surgiram antes que o módulo pudesse ser liberado para a Prefeitura de São Paulo e consumido com estabilidade por todos os perfis.

solicitação do negócio

Fernando (Coord. Planejamento SP) pediu que usuários da prefeitura vejam apenas equipamentos ativos.

Antes de liberar o Radar para a Prefeitura de São Paulo, era necessário garantir que dados de equipamentos inativos não fossem expostos. O filtro de status (Ativo/Inativo) precisava ser removido e a consulta forçada para active=1.

observação de produção

O dashboard ficava em branco enquanto o cache aquecia em produção.

Na primeira carga, sem cache disponível, o dashboard levava tempo para retornar os dados. O usuário ficava olhando para uma tela vazia, sem nenhum indicador de que algo estava acontecendo.

Filtro de Status removido para a Prefeitura.

O backend agora detecta usuários municipais e força active=1 em todas as consultas. O frontend apenas reflete a capacidade que o backend entrega.

Problema

Prefeitura via equipamentos inativos.

O filtro de Status (Ativo/Inativo) era universal; qualquer perfil podia alternar e ver dados inativos.

→
Risco

Dados sensíveis expostos.

Fernando (Coord. Planejamento SP) vetou a liberação enquanto inativos fossem acessíveis. Bloqueio total do rollout prefeitura.

→
Solução

Backend força active=1.

Detector shouldForceActiveOnly() via can('municipal', Ticket). Filtro de Status some do frontend quando backend omite active_options.

Detector municipal no RadarRepository

private function shouldForceActiveOnly(): bool { return Auth::check() && Auth::user()->can('municipal', Ticket::class); }

Forçando active=1 no pipeline de filtros

if ($this->shouldForceActiveOnly()) { $query->where('places.active', 1); } elseif ($request->filled('active')) { $query->where('active', $request->get('active')); }

O que o backend garante.

  • active_options omitido dos filtros para municipal
  • active=0 em query string é ignorado (forçado para 1)
  • Detail e popup de equipamento inativo retornam 404
  • Popup de mapa agora passa por radar-mapa permission scope
  • Cache keys incluem is_municipal — separação automática

O que o frontend reflete.

  • Filtro de Status some com v-if="options.active_options && options.active_options.length"
  • active_options retorna null com segurança (?? null)
  • Nenhuma lógica de permissão no frontend — backend é a fonte da verdade
Compatibilidade entre concessões

A implementação não é hardcoded para SP. O detector usa manutencao-prefeitura como alias de permissão. Qualquer usuário prefeitura em qualquer praça (BA, RJ, MG, CE, etc.) com permissões radar-* recebe automaticamente:

  • Forçamento de active=1 em todos os endpoints
  • Filtro de Status oculto no frontend
  • Cache keys separadas via contexto is_municipal

Nenhuma alteração por concessão é necessária.

Ativos primeiro, inativos depois.

O grid de equipamentos agora ordena ativos antes de inativos, e a página de equipamentos inicia filtrada em "Ativo" por padrão.

ordenação

Active DESC no backend.

A query de cards agora usa orderBy('places.active', 'desc')->orderBy('places.name'). Todos os usuários veem equipamentos ativos primeiro, independentemente de filtros. Isso é um comportamento global, não apenas para prefeitura.

default de UX

Primeira carga mostra só Ativos.

Na primeira visita à página de equipamentos, se o usuário tem o filtro de Status disponível e não definiu nada, o frontend automaticamente seleciona "Ativo". Limpar filtros volta a mostrar tudo. A lógica vive no componente, não na store.

Ordenação no RadarRepository

$idsQuery->orderBy('places.active', 'desc') ->orderBy('places.name');

Default ativo no componente

if (!this.defaultActiveApplied && !params.active) { params.active = '1'; }

Skeleton de carregamento no Dashboard.

Substituição do spinner global por um skeleton local que espelha o layout real do dashboard. Feedback imediato ao usuário enquanto o cache aquece.

antes

Tela em branco.

Na primeira carga em produção, sem cache disponível, o dashboard exibia uma tela completamente vazia por vários segundos. O usuário não sabia se algo estava acontecendo ou se a página estava quebrada.

agora

Skeleton com layout do dashboard.

Um skeleton com a estrutura exata do dashboard (KPIs, gráficos, tabela) aparece imediatamente. Quando os dados chegam, o skeleton é substituído pelo dashboard real com animação. Em caso de erro, um estado de retry é exibido.

Decisões de implementação.

  • Skeleton local em vez de spinner global — evita conflitos com outros componentes que usam #spinner-global
  • Estado controlado por hasLoaded + loadFailed, independente do Vuex loading
  • Skeleton renderiza no primeiro paint (showSkeleton = !hasLoaded && !loadFailed)
  • Apenas remove quando dados resolvem com sucesso (hasLoaded = true)

Estados cobertos.

  • Carregamento inicial: skeleton com KPIs, gráficos e tabela fantasma
  • Sucesso: skeleton some, dashboard anima entrada
  • Erro de rede: estado de erro com mensagem e botão "Tentar novamente"
  • Vazio: mensagem "Nenhum resultado para os filtros selecionados" + botão "Limpar filtros"
  • Requisições concorrentes: dashboardRequestId descarta respostas de requisições antigas

Controle de estado no RadarDashboard.vue

showSkeleton: !this.hasLoaded && !this.loadFailed // Skeleton aparece imediatamente, some só com dados reais showEmptyState: this.hasLoaded && !this.loading && (this.loadFailed || this.isEmpty) // Erro ou vazio = estado tratado, não tela em branco

Como cada decisão foi tomada.

O fluxo que levou da solicitação à implementação.

Filtro Prefeitura

Backlog → Implementação

1. Solicitação do negócio

Fernando (Coord. Planejamento SP) solicita: prefeitura não pode ver inativos antes da liberação.

2. Análise técnica

Detector via can('municipal', Ticket) — mesmo padrão de 85+ call sites no repo. Backend como fonte da verdade.

3. Implementação

Force active-only no RadarRepository. Frontend reflete capacidade do backend. Cache keys automáticas.

Sorter Ativo

Solicitação → Revisão

1. Solicitação

"Para perfis com filtro de status, sempre mostrar ativos primeiro no grid de equipamentos."

2. Implementação

Ordenação active DESC, name no backend + default active=1 no frontend na primeira carga.

3. Revisão técnica

Removeu magic flag _applyDefaultActive da store. Lógica movida para o componente. Corrigido double-fetch em clearAll().

Dashboard Skeleton

Observação → Implementação

1. Observação em produção

Dashboard sem cache mostrava tela em branco. Usuário sem feedback visual.

2. Decisão de UX

Skeleton local em vez de spinner global: evita conflito com outros componentes e dá contexto visual do layout.

3. Implementação

Skeleton CSS puro com animação de sweep. Estados de erro e vazio com ações (retry, limpar filtros).

O que precisa acontecer no deploy.

Passos para o rollout seguro em produção.

1. Deploy do código

Merge do branch radar-hotfixes em main. Nenhuma migration ou nova dependência.

2. Build de assets

npm run production para compilar o novo app.js com skeleton e filtro condicional.

3. Conceder permissões

Atribuir radar-equipamentos, radar-dashboard, radar-mapa ao perfil prefeitura.

4. Invalidar sessões

Usuários prefeitura devem refazer login — permissões são cacheadas na sessão.

Notas sobre cache
  • Primeiras requisições após deploy serão cache misses para todos os usuários — o contexto is_municipal é novo nas cache keys.
  • Isso é seguro e auto-regenerativo. Nenhuma limpeza manual de cache é necessária.
  • Se o filtro de Status ainda aparecer após deploy, é cache de browser (Ctrl+Shift+R resolve).

Três hotfixes, zero breaking changes.

Sem migrations, sem novas dependências, sem alterações de config. Apenas código que reforça invariantes de segurança, ordenação intuitiva e experiência de uso. O Radar está pronto para a Prefeitura de São Paulo.
