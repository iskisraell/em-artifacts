Arquitetura.

O pipeline segue o padrão Command → Job — o Command agenda, os Jobs processam e a persistência vai para as tabelas dedicadas.

1. Command

Agendamento

saia:analyze

Lê a config, aplica feature flags e safety window, despacha jobs em batch.

2. Jobs

Processamento

SaiaAnalyzeTicket

Busca atendimentos SP ativos, monta prompt Gemini, persiste resultado com hash.

3. Persistência

Resultado

saia_analysis_runs + items

Cada execução gera um Run. Cada ticket analisado gera um Item com hash idempotente.

Validação.

Agent loop de 3 camadas: implementação, verificação estática e revisão adversarial por agente Codex.

O loop

  1. Layer 1 — OpenCode: implementou os 5 arquivos seguindo o design aprovado.
  2. Layer 2 — Verificação: php artisan migrate --pretend, lint, typecheck no PHP 7.4.
  3. Layer 3 — Codex: revisão adversarial encontrou 1 problema real (P1).

P1 encontrado e corrigido

nullable em índice único MySQL. A migration saia_analysis_items definia uma coluna como nullable mas com índice único — o MySQL trata múltiplos NULLs como distintos, o que quebraria a premissa de idempotência por hash.

Corrigido: coluna ticket_id passou a NOT NULL com default 0, garantindo unicidade real.

Próximos passos.

Slice 2 foca em executar sem efeito colateral: dry-run completo com o hash guard ativo.

Slice 2 — Job Dry-Run

Execução real do job sem persistência, validando prompt, latência e contrato de resposta do Gemini.

Slice 2 — Hash Guard

Verificação de idempotência antes de inserir: se o hash já existe, pula. Garante reprocessamento seguro.
