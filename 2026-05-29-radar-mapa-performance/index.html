Contexto.

De onde viemos e por que isso importa agora.

Antes

  • Mapa travava o browser por ~6s ao carregar 6k pontos (Supercluster KD-tree na thread principal)
  • Marcadores na mesma coordenada se sobrepunham, só o topo era clicável
  • Cada termo de busca criava uma entrada de cache de 6 horas no filter-options
  • Dashboard e mapa disparavam em sequência, somando latência
  • Query do mapa fazia LEFT JOIN em place_shelter_informations para cada ponto

Depois

  • Supercluster roda em Web Worker dedicado, browser continua pintando frames
  • Pontos co-localizados viram badge com contagem + carousel de popups
  • Busca textual ignora cache de filter-options (cross-filtering preservado)
  • Dashboard e mapa disparam em paralelo via Promise.all
  • Query do mapa retorna só 7 campos, sem JOIN, via raw builder

Web Worker.

O ganho de performance mais significativo: Supercluster off main thread.

Arquitetura

Thread Principal (RadarMap.vue)

Leaflet map rendering DOM management (≤150 markers) Popup lazy-fetch fitBounds / setView

Worker (radar-cluster.worker.js)

Supercluster.load() -- KD-tree build getClusters() getLeaves() getClusterExpansionZoom()

Comunicação via postMessage / onmessage. O worker é compilado como entry separado no webpack (31.5 KiB, self-contained com kdbush). O mapa continua respondendo enquanto o índice é construído.

Lazy Popup Endpoint

Novo endpoint GET /radar/data/map/popup/{place} carrega dados do popup sob demanda (nome, sync_id, stop_number). O popup abre com spinner e atualiza quando a resposta chega.

Dashboard e mapa agora disparam em paralelo:

// Antes: sequencial await fetchDashboard(); await fetchMapData(); // Depois: paralelo await Promise.all([ fetchDashboard(), fetchMapData() ]);

Limite padrão do mapa: 15.000 → 50.000 pontos (seguro com worker off main thread).

Agrupamento de pontos.

De marcadores sobrepostos para carousel navegável.

1. Grouping

applyGroupedMarkers() agrupa features por coordenada arredondada para 5 decimais (~1.1m precisão). Grupos de 1 ponto: marker normal. Grupos de 2+: badge laranja com contagem.

2. Co-location

Clusters pequenos (≤8 pontos) verificam co-localização antes de spiderfyar. Se todos compartilham coordenada: abre carousel diretamente. Se não: spiderfy normal.

3. Carousel

Barra de navegação ← 1 de 3 → com dots indicadores. Cada slide carrega dados via endpoint lazy. Footer "Ver detalhes" abre drawer de detalhe.

Cache inteligente.

Por que search não deve ser cacheado em filter-options.

O problema do cache com busca

Cada termo de busca único gerava uma entrada separada de cache com TTL de 6 horas. O keyspace era ilimitado: escopo × filtros × termo_busca.

Termos de busca são efêmeros. Um usuário digita "rest", depois "resta", depois "restau". Cada uma criava uma entrada de cache que ninguém mais ia usar.

A solução: bypass total

Quando $request->filled('search'), o getFilterOptions() retorna direto via computeFilterOptions(), sem passar pelo rememberHeavyPayload.

O cross-filtering (dropdowns se adaptam ao resultado da busca) é preservado porque o termo de busca ainda flui para filterOptionPlaceIds(). A única diferença é que o resultado não é cacheado.
