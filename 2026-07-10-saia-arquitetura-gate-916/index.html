O que esta validado e o que ainda nao foi implementado.

Fundacao dry-run entregue em codex/saia-maintenance-analysis-foundation. Contrato Gemini testado em smoke isolado. Pipeline live ainda e desenho proposto.

validado

Fundacao dry-run completa.

  • Config config/saia.php com provider, mode, safety window, feature flags, establishment/activity scope, profile key/version.
  • Migrations 4 tabelas: saia_analysis_runs, saia_ticket_analyses, saia_analysis_run_items (ledger), mais coluna dispatched_count.
  • Models SaiaAnalysisRun, SaiaTicketAnalysis com casts JSON, relationships e finalizeIfComplete().
  • Evidence Resolver MaintenanceTicketEvidenceResolver: opening, closing, problem photos com fontes documentadas.
  • Profile maintenance_dry_run.json v1 com 21 termos de glossario, 8 regras, schema esperado.
  • Prompt Builder ProfilePromptBuilder — carrega profile, valida, monta prompt PT-BR com placeholders de evidencia.
  • Hash idempotencia SHA-256 canonico sobre evidencia ordenada.
  • Command saia:analyze-maintenance com cache lock, query de candidatos, dispatch por chunk.
  • Job SaiaAnalyzeMaintenanceTicketJob com tries=3, backoff=[60,120,300], persistencia dry-run.
  • Schedule Kernel.php: hourlyAt(13) com withoutOverlapping(120).
  • Smoke Gemini REST isolado no container: SAIA_SMOKE_OK, finishReason=STOP, modelo confirmado.
  • Testes 23 testes Feature cobrindo idempotencia, hash, contagem de run, config, backfill de prompt.
pendente

Pipeline live + HITL nao implementados.

  • Gemini multimodal — envio de imagem do ticket junto com prompt textual. API Gemini Vision nao testada.
  • Normalizacao da resposta — parser do JSON bruto do Gemini para o expected_output_schema do profile.
  • Persistencia raw/normalized — colunas raw_response e normalized_response existem mas nunca sao populadas.
  • Confidence scoring — coluna confidence existe mas nunca e calculada.
  • UI HITL (#919) — zero codigo Vue. Nao existe tela de revisao, fila, acoes humanas.
  • Checklist answers no evidence — resolver so coleta fotos. Respostas de checklist (ticket_answers) nao entram no pacote de evidencia.
  • Conclusion photos multiplas — resolver pega apenas a primeira closing photo. Demais closing + auxiliary closing nao sao incluidas.
  • Geographic scope RJ — config tem feature_saia_rj_enabled mas nao ha establishment IDs do RJ.
  • Eligibility pre-Gemini — nao existe validacao se o ticket tem evidencia suficiente antes de chamar Gemini (custo).
  • Politica de retry live — backoff do job cobre falha de infra, mas nao ha estrategia para falha do modelo (timeout, rate limit, resposta invalida).

Como o profile entra no prompt.

ProfilePromptBuilder::build() monta um prompt PT-BR com: {equipment} = lista de equipment_types {glossary} = tabela termo | definicao | severidade {rules} = regras numeradas {schema} = expected_output_schema como JSON {evidence} = ticket_id + paths das fotos Prompt pronto e salvo em generated_prompt no campo evidence do SaiaTicketAnalysis.

Fluxo Ticket ate HITL, distinguindo dry-run atual do live proposto.

Use as abas para alternar entre o estado atual (100% codado) e o desenho proposto para o pipeline live.

1. Query tickets

Command seleciona tickets SP, status IN (completed, completed_by_system), activities [27,28,11,32], estab [383,384], safety window 6h.

2. Evidence resolve

Resolver coleta opening, closing, problem photos. Monta pacote de evidencia deterministico.

3. Hash + Prompt build

SaiaTicketAnalysis com status=completed_dry_run. Ledger em saia_analysis_run_items. Run finalizado quando todos processados.

estado atual

O que roda hoje.

O job nao chama Gemini. Todo o fluxo para na persistencia dry-run. O prompt e gerado e salvo, mas nunca enviado. Nao ha consumo de creditos, nao ha risco de resposta invalida. A fundacao prova que o pipeline de evidencias funciona e que o hash de idempotencia detecta late mirroring.

1. Query tickets

Igual dry-run. Candidate tickets com verificação de eligibility (evidencia suficiente?).

2. Evidence resolve

Ampliado com checklist answers, todas as closing photos. Hash de idempotencia.

3. Prompt build + Gemini

Prompt com imagem via Gemini Vision API. Modelo gemini-flash-lite-latest. Timeout, retry, rate limit.

4. Validate + Normalize

Parser do JSON bruto contra expected_output_schema. Fallback para resposta invalida.

etapa 5

Persist dual.

raw_response (JSON bruto do Gemini) + normalized_response (schema validado) + confidence. status = analyzed ou needs_review.

etapa 6

Ledger + Run finalize.

Igual dry-run: saia_analysis_run_items atualizado. Run counters incrementados. Run finalizado quando batch completo.

etapa 7

UI HITL #919.

Tela de revisao humana: foto lado a lado com diagnostico do Gemini. Acao humana: confirmar, rejeitar, solicitar reanalise. Zero codigo hoje.

proposto

Diferencas criticas do live.

O live troca completed_dry_run por chamada real ao Gemini, com imagem + prompt. Todo o resto (hash, persistencia, ledger, finalizacao de run) e reutilizado. A separacao dry-run/live e controlada por config('saia.analysis_mode') — default dry_run, live via env SAIA_ANALYSIS_MODE=live. Nenhuma decisao automatica: o sistema nunca altera status de ticket. Revisao humana e obrigatoria antes de qualquer acao.

Contrato seguro proposto para o pipeline live.

Principios de design que protegem contra gasto desnecessario, corrupcao de dados e decisoes automaticas.

Idempotencia por hash.

SHA-256 sobre evidencia canonica ordenada com recursiveKsort. Unique constraint em (ticket_id, profile_key, profile_version, input_hash). Ticket com mesma evidencia = nunca reanalisado (sem --force).

implementado SHA 441f2ca85

Dry-run / Live separation.

analysis_mode em config/saia.php. Default dry_run. Live so ativado com env SAIA_ANALYSIS_MODE=live. Impossivel chamar Gemini acidentalmente em producao.

implementado SHA 6be6d3a9b

Raw + Normalized dual.

raw_response preserva o JSON exato do Gemini (auditavel). normalized_response e o parsing validado contra expected_output_schema. Separacao permite re-parse sem re-chamar Gemini.

schema existe, logica pendente

Fail & Retry.

  • Job retry: tries=3, backoff=[60, 120, 300] no queue scripts.
  • Timeout Gemini: config('saia.gemini_timeout') = 30s default.
  • Falha de modelo: resposta invalida, timeout, rate limit — job falha, failed() notifica Slack.
  • Pendencia: estrategia de retry para Rate Limit (429) e resposta JSON invalida ainda nao existe. gate #916

Confidence & Limits.

  • Threshold: confidence_threshold_needs_review (0.7 default). Abaixo = status=needs_review.
  • Safety window: 6h apos end_date antes de analisar. Evidencia incompleta = ticket excluido.
  • Batch limit: analysis_batch_size = 50 tickets por run. Evita sobrecarga.
  • Photo limit: evidence_photo_limit = 10. Protege prompt de estourar token limit.
  • Nenhuma decisao automatica: saIA nunca altera tickets.status. Revisao humana obrigatoria.
regra de ouro

O sistema diagnostica; o humano decide.

saIA produz um diagnostico estruturado com confianca. Nenhum ticket e fechado, reaberto, reclassificado ou alterado automaticamente. O HITL (#919) e o unico caminho para acao. Esse principio e irrevogavel e deve ser mantido em todas as fases.

Por que esta arquitetura e boa — e por que ainda nao e suficiente.

Avaliacao honesta dos trade-offs com recomendacao condicionada aos gates.

pontos fortes

O que funciona bem.

  • Hash idempotency — protege contra reanalise e late mirroring. Testado com 20+ cenarios.
  • Profile versionado — prompt e revisado em PR. Glossario e regras evoluem sem deploy de codigo.
  • Dry-run default — impossivel gastar creditos sem explicitamente ativar SAIA_ANALYSIS_MODE=live.
  • Reuso de padrao Gemini — mesmo GeminiService usado pelo Posting AI. Smoke confirmado.
  • Ledger independente — saia_analysis_run_items sem FK em ticket. Audit trail preservado.
  • Separacao raw/normalized — permite evoluir o parser sem re-chamar Gemini.
  • Nao altera tickets — zero risco de corrupcao de dados de producao.
riscos & limitacoes

O que preocupa.

  • Multimodal nao testado — Gemini Vision com foto de abrigo nunca rodou. Nao sabemos qualidade, latencia ou custo.
  • Modelo unico — gemini-flash-lite-latest e fixo no codigo. Mudanca exige deploy. Sem fallback para outro provider.
  • Evidencia incompleta — sem checklist answers, sem todas as closing photos. Diagnostico pode perder contexto.
  • Normalizacao fragil — parser de JSON de LLM e notoriamente instavel. Precisa de validacao + fallback.
  • Sem eligibility gate — ticket sem foto de abertura ainda sera dispatcheado. Custo desperdicado.
  • HITL e futuro distante — sem UI, o pipeline live gera dados que ninguem ve. Valor de negocio zero ate #919.
  • Custo invisivel — sem monitoramento de tokens/gastos por run. Orcamento nao controlado.

Recomendacao condicionada.

Prosseguir com #916 (Gemini live) somente apos: (1) teste multimodal com 10 tickets reais e metrica de qualidade, (2) parser de normalizacao com fallback, (3) eligibility gate que pula tickets sem foto de abertura. Adiar HITL #919 ate que o diagnostico tenha qualidade minima validada — nao faz sentido construir UI para dados ruins.

condicional

Decisoes que o Israel precisa aprovar antes de codar live.

Cada decisao abaixo bloqueia o proximo passo. Ordenadas por precedencia.

1. Multimodal — Gemini Vision ou outro provider?
O Gemini suporta imagem inline via API generateContent com inlineData. Alternativa: enviar so texto, depender de descricao previa. Custo e latencia da abordagem multimodal precisam ser medidos.
#916
2. Contrato de normalizacao — schema fixo ou flexivel?
O expected_output_schema do profile define a estrutura esperada. O parser deve ser estrito (rejeitar JSON fora do schema) ou leniente (extrair o que conseguir)? Estrito = mais seguro, leniente = mais dados.
#916
3. Eligibility — quais tickets merecem analise?
Sem foto de abertura = skip? Sem closing = skip? So opening = analisar so com abertura? Cada cenario tem trade-off de cobertura vs custo.
#916
4. Ampliar evidence — checklist answers e mais fotos?
Incluir ticket_answers aumenta custo do prompt mas da contexto valioso. Incluir todas as closing photos melhora diagnostico pos-manutencao. Escopo deve ser decidido.
#916
5. Retry policy para falha de modelo — quando desistir?
Rate limit (429) deve ter retry com backoff. JSON invalido do modelo deve tentar re-prompt ou so logar? Timeout deve tentar com modelo mais leve?
#916
6. Monitoramento de custo — como rastrear gastos?
Sem usageMetadata no ledger, nao sabemos quanto cada run custa. Adicionar prompt_tokens, completion_tokens, cost_estimate ao SaiaAnalysisRun ou ao SaiaTicketAnalysis?
#908
7. RJ scope — quando ativar?
feature_saia_rj_enabled existe mas nao ha establishment IDs. SP precisa estar estavel primeiro. Definir gate de qualidade minima para expandir.
