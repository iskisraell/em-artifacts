Antes vs Agora

Do reativo de 3 campos ao preventivo com 30+ campos de diagnóstico.

4YouSee

  • 3 campos: place_sync_id, sync_id, status="offline"
  • Reativo: só quando player cai
  • Sem diagnóstico embutido
  • Sem idempotência formal

IoT Hub

  • 30+ campos de telemetria contextual
  • Preventivo: alertas antes da queda
  • Diagnóstico embutido: térmico, elétrico, conectividade
  • Outbox + idempotência

Decisão de Arquitetura

Seis métodos avaliados. Webhook com outbox lidera por esforço, latência e reversibilidade.

Webhook — Recomendado

Nota 35/40. Segue o padrão 4YouSee já validado. Outbox garante entrega mesmo quando Operações está fora; idempotência evita duplicatas. Evolução futura para RabbitMQ/SQS preserva o mesmo payload.

Message Queue (RabbitMQ/SQS)

Nota 31/40. Melhor transporte tecnicamente, mas setup e operações pesados para um MVP. Gatilhos para migração: >1.000 alertas/dia, bidirecionalidade ou SLA 99,99%.

API Consumer

Nota 29/40. Faz o IoT Hub conhecer demais os internals do Operações e do package PHP. Aumenta acoplamento sem ganho claro de latência ou confiabilidade.

Polling

Nota 28/40. Simples e reversível, mas latência alta e burst handling fraco. Não escala bem para alertas em tempo real.

Mapeamento de Dispositivos

IoT Hub usa SEP e localidade; Operações usa place.sync_id e endereço. A correção do ticket depende do mapeamento.

Desafio

Dois vocabulários de lugar

IoT Hub identifica por dispositivo.sep e localidade (cidade/UF). Operações usa place.sync_id (legacy Building.cod) com lat/lon, bairro e square.

Regra

Sem fuzzy na produção

Ticket só é criado para mapeamento validado. Alertas não mapeados viram mapping_pending e notificam a equipe de integração.

Prioridade Método Uso
1 Mapeamento manual aprovado Verdade de produção
2 SEP ↔ place.sync_id Só se SEP = legacy cod for comprovado
3 Player.sync_id Se houver conexão entre os sistemas
4 Coordenadas + endereço Apenas sugestão; nunca auto-criar ticket
5 Cidade/localidade fuzzy match Inaceitável para automação

Fases de Implementação

MVP unidirectional, hardening operacional e ciclo de vida bidirecional.

Phase 0

Contract Alignment

Thomas define payload e regras ticketable. William confirma campos obrigatórios, prioridade e semântica de duplicados.

Phase 1

MVP Unidirectional

Outbox no IoT Hub; endpoint autenticado no Operações; job cria ticket com nota de telemetria e audit table.

Phase 2

Hardening

Métricas, debounce de flapping, fila mapping_pending e notificação Slack para alertas críticos não mapeados.

Phase 3

Bidirecional

Operações emite ticket resolved/reopened; IoT Hub decide se resolve alerta ou apenas anota.

Top 5 Riscos

O que pode dar errado e como mitigar.

1. Mapeamento errado cria ticket no local errado

Tabela de mapeamento com aprovação manual; sem auto-fuzzy na produção.

Muito alto

2. Tempestade de tickets duplicados no flapping

Idempotency key, open-ticket guard e debounce window por alerta.

Muito alto

3. Perda silenciosa de mensagens

IoT outbox, retry state, replay command e delivery dashboard.

Alto

4. Bypass de segurança via endpoints IoT sem auth

Apenas alertas server-side emitem eventos; Passport + HMAC no webhook.

Alto

5. Ticket note vira dump de telemetria ilegível

Resumo conciso na nota; payload completo na tabela de auditoria.

Conclusão

Webhook com outbox e idempotência é o MVP mais maduro: reusa o padrão 4YouSee, entrega em segundos e evolui para RabbitMQ/SQS quando o volume exigir.
